## [Unreleased]

### Added
- **LTC audio-to-timecode conversion**: timeline clips can now decode Linear Timecode (LTC) from their source audio and store the result in the existing `source_timecode_base_ns` metadata path, so converted clips immediately participate in **Align Grouped Clips by Timecode** and newly placed clips from the same source reuse that timecode metadata.
  - **Timeline UI**: right-click a single source-backed audio/video clip and choose **Convert LTC Audio to Timecode…**. The dialog supports `Auto Detect`, `Left Channel`, `Right Channel`, and `Mono Mix`, plus an optional frame-rate override (`23.976`, `24`, `25`, `29.97`, `30`).
  - **Audio cleanup**: when LTC lives on one stereo side, UltimateSlice automatically routes the opposite side to both speakers using the existing `AudioChannelMode::{Left, Right}` behavior so the timecode tone is no longer audible. Mono-only LTC clips are muted after conversion.
  - **Persistence / placement**: the decoded base timecode is written back to the matching library item, the active source-monitor selection, and every same-source clip in the project, so save/load, grouped timecode alignment, append/insert/overwrite, and timeline drag/drop all stay consistent.
  - **MCP**: new `convert_ltc_audio_to_timecode` tool exposes the same workflow for automation and returns the decoded timecode, resolved LTC channel, updated clip count, and resulting audio-routing action.
  - **Tests**: new LTC decoder unit coverage verifies generated-frame decoding, source-base rebasing with leading silence, audio-routing decisions, placement propagation, and recursive same-source clip updates.
- **7-band match EQ for fine mic-matching** (audio matching improvement): the audio match engine now produces a 7-band EQ in addition to the existing 3-band EQ, allowing much finer corrections when matching one mic's tonal balance to another (e.g., making a lav mic sound more like a shotgun mic). The 7 bands cover ~100 Hz, 200 Hz, 400 Hz, 800 Hz, 2 kHz, 5 kHz, and 9 kHz — each mapped from the 11-band spectral analysis to address body/clothing resonance, chest/proximity, low-mid muddiness, fundamental speech, presence, and air. The match EQ is stored as `clip.match_eq_bands: Vec<EqBand>` independently of the user-facing 3-band EQ; both are applied in series during export, with the match EQ first. The 3-band EQ continues to be used for manual user adjustments.
  - **Inspector UI**: a small frequency-response curve below the **Match Audio…** button visualizes the active 7-band correction (log-frequency X axis, ±12 dB Y axis, filled curve with band markers). A **Clear Match EQ** button (visible only when match EQ is set) resets the 7-band match without touching the user 3-band EQ.
  - **Live preview**: the program player wires a dedicated `equalizer-nbands` element into each slot's audio chain (linked before the user EQ). Updating match EQ on the same band count is in-place (no rebuild); changing the band count triggers a slot rebuild. The audio-only multi-pipeline path also creates a per-branch match equalizer.
  - **Export**: a `build_match_eq_filter()` FFmpeg filter chain is inserted before the user EQ in all three audio assembly sites (primary video clips, secondary video clips, audio-only clips).
  - **MCP**: new `clear_match_eq` tool resets just the match EQ; the `match_clip_audio` response now includes `match_eq_bands` alongside `eq_bands`.
  - **Persistence**: round-trips through FCPXML via the `us:match-eq-bands` custom attribute (writer + parser) AND through OTIO via new `eq_bands` and `match_eq_bands` fields on `UltimateSliceClipOtioMetadata`. The OTIO addition also closes a pre-existing gap — the user 3-band EQ (`clip.eq_bands`) was not previously serialized to OTIO at all, so this completes the audio EQ persistence story symmetrically across both interchange formats.
  - **Tests**: 3 new audio match unit tests covering the 7-band output (lav→shotgun shape, deadzone, band count), an extended undo test covering the new match-eq fields, and 2 new OTIO metadata tests (round-trip of both EQs + verification that empty match-eq is omitted from JSON).

### Added
- **AI frame interpolation for slow-motion clips (RIFE)**: a new `Slow-Motion Interpolation → AI Interpolation (RIFE)` option in the Inspector's Speed section. When a clip has `speed < 1.0` somewhere on its curve and AI mode is enabled, a new background `FrameInterpCache` (modeled on `BgRemovalCache`) precomputes a higher-fps sidecar by running the source through a RIFE ONNX model pairwise (multiplier `M = ceil(1 / min_speed)` clamped to 2–8×) and encoding the interpolated frames into an H.264 mp4 sidecar at `$XDG_CACHE_HOME/ultimateslice/frame_interp/<hash>.mp4`. Both Program Monitor preview and FFmpeg export consume the sidecar at the source-path resolution layer (alongside `bg_removal_paths` / `proxy_paths` / `animated_svg_paths`), so the visible frames match exactly between preview and export — no double interpolation, since the existing `minterpolate` filter is skipped in AI mode. The cache uses a 2-thread worker pool, sidecar identity keyed by `(source_path, multiplier)`, atomic `.partial → final` rename, graceful fallback to a CPU blend on RIFE inference failure, and the same `find_model_path()` discovery pattern as MODNet (executable-relative, dev `data/models/`, Flatpak `/app/share/`, XDG `~/.local/share/ultimateslice/models/rife.onnx`). Inspector exposes a 4th dropdown entry plus a status row label, Preferences → Models gains a RIFE entry with an installation hint and clickable model directory link, and the new MCP `set_clip_speed` tool sets `speed` and `slow_motion_interp` (`"off"` / `"blend"` / `"optical-flow"` / `"ai"`) atomically and queues the AI sidecar generation when applicable. FCPXML / `.uspxml` round-trip the new variant via the existing `us:slow-motion-interp` vendor attr (string `"ai"`). The `SlowMotionInterp::has_slow_motion()` and `min_effective_speed()` helpers were promoted from private to public so the cache can size the multiplier from outside the model crate. RIFE is not bundled — users install it manually like MODNet/MusicGen/Whisper. Five new unit tests in `frame_interp_cache` cover multiplier selection across speed values and the request-gating predicate.
- **Remove Silent Parts dialog now has a Suggest button**: the **Right-click → Remove Silent Parts…** dialog gains a **Suggest** button next to the silence-threshold spinner. Clicking it runs the same ffmpeg `astats` noise-floor analysis used by the Inspector's voice-isolation Suggest button (5th-percentile windowed RMS + 6 dB headroom, clamped to −60…−10 dBFS) and writes the result back into the spin button, with a "Suggested: −N dBFS" status label below. Runs synchronously on the main thread (typically 1–3 s for short clips); the button briefly disables itself during analysis and the dialog pumps pending GTK events first so the "Analyzing…" status paints before the blocking call. Helps users who don't want to guess at a threshold for noisy footage.
- **Voice isolation can now operate without subtitles**: voice isolation previously hard-required Whisper subtitles to find speech regions — clips that didn't transcribe well (music beds, ambient/B-roll, foreign-language clips) had no isolation option at all. The Inspector's Voice Isolation section now exposes a **Source** dropdown with two modes: **Subtitles** (existing Whisper-word-timing path, unchanged default) and **Silence Detect** (new). In Silence Detect mode, the inspector reveals a **Silence threshold** slider (−60 to −10 dB, default −30), a **Min gap** slider (50 to 2000 ms, default 200), and two new buttons: **Suggest** runs ffmpeg `astats` over the clip and auto-picks a threshold from the 5th-percentile windowed RMS plus 6 dB headroom (a robust noise-floor estimate that ignores intro/outro silences and loud transients), and **Analyze Audio** runs ffmpeg `silencedetect` with the current threshold/min-gap and stores the inverted speech intervals on the clip. A **Speech intervals: N** label shows the cached count, or "Not analyzed" when no analysis has been run. Threshold/min-gap edits invalidate the cache so the user must re-Analyze. The unified `Clip::voice_isolation_speech_intervals_ns` helper feeds both real-time preview (`program_player::volume_at_timeline_ns`) and export (`build_volume_filter`), so silence-mode and subtitle-mode follow exactly the same pad/merge/duck/cosine-fade math — no path-specific code. The cached intervals are NOT serialized in FCPXML (the source mode + threshold + min-gap settings are, via three new `us:voice-isolation-*` vendor attrs); on project reload the user re-clicks **Analyze Audio**, mirroring the existing "Generate Subtitles" UX. Four new MCP tools (`set_voice_isolation_source`, `set_voice_isolation_silence_params`, `suggest_voice_isolation_threshold`, `analyze_voice_isolation_silence`) and three new undo commands (`SetClipVoiceIsolationSourceCommand`, `SetClipVoiceIsolationSilenceParamsCommand`, `AnalyzeVoiceIsolationSilenceCommand`) round out the feature. The Suggest button does not chain into Analyze automatically — it just sets the slider — so the user can review the suggestion before running the (~1–3 s) silencedetect pass.
- **Generate Music: optional reference-audio style guidance**: the Generate Music dialog now exposes a **Choose Reference Audio…** picker that runs lightweight pure-Rust feature extraction (BPM via spectral-flux autocorrelation with octave-correction smoothing, key/mode via Krumhansl-Schmuckler chroma matching, brightness via spectral centroid bucketing, dynamics via per-frame RMS coefficient-of-variation) on the first 30 seconds of any audio or video file. The derived natural-language descriptors (e.g. *"around 128 BPM, in the key of C major, bright timbre, dynamic energy"*) are prefilled into an editable Style Hints field and appended to the user's text prompt before the existing musicgen-small inference loop runs. Analysis runs on a background thread with generation-counter staleness handling so picking a different file mid-analysis discards the in-flight result. The MCP `generate_music` tool gains a matching optional `reference_audio_path` argument that performs the same analysis synchronously inside the main-thread handler with graceful degradation — analysis failures log a warning and the tool succeeds with the original prompt. New `src/media/audio_features.rs` module hosts the analysis (reusing `audio_sync::extract_mono_audio_samples` for GStreamer-backed PCM decoding and the existing `rustfft` dependency for FFTs), with seven unit tests covering sine/click-train/noise/silence inputs and the prompt-hint formatter. The MusicGen model itself is unchanged: musicgen-melody (the only variant with native audio conditioning) currently has no off-the-shelf ONNX export, so this ships as text-prompt augmentation on top of musicgen-small, leaving a clean upgrade path if a melody ONNX becomes available.

- **Position sliders now allow off-canvas placement (range expanded from ±1.0 to ±3.0)**: Inspector Position X/Y sliders, the transform-overlay pan handle, keyboard nudges, the MCP `set_clip_transform` clamp, the model-level keyframe clamp, and the direct-canvas-translation math used by titles, adjustment scopes, and tracker-followed clips all accept positions past the canvas edge. ### Fixed
- **Editing a subtitle segment now updates the preview and export immediately, including word highlight / karaoke mode**: the program monitor's subtitle overlay collection was nested inside the `if pos_ns != last_pos_ns_c.get()` poll-tick gate, so subtitle text edits made while paused never showed up until the playhead moved. The subtitle inspector also only committed via Enter / focus-out, so typing then jumping straight to Export left the model holding the un-committed text. The overlay collection now runs every poll tick (cost is negligible — `proj.tracks` is already borrowed for the surrounding inspector poll), and the subtitle text entry has a `connect_changed` handler that mutates `clip.subtitle_segments[i].text` on every keystroke (matching the existing pattern used by the title text entry), with Enter / focus-out still pushing a single `EditSubtitleTextCommand` per typing session so undo collapses the whole edit instead of one entry per keystroke. Word highlight / karaoke mode reads from `seg.words[i].text` instead of `seg.text`, and the old code only updated `seg.text`, so karaoke kept showing the original Whisper tokens after an edit. Both edit paths (`connect_changed` and `EditSubtitleTextCommand::execute`/`undo`) now go through a new `SubtitleSegment::set_text_and_resync_words` helper that replaces each word's text in place when the new whitespace-split count matches the old word count (preserving Whisper's per-word timings), and otherwise redistributes the segment's `start_ns..end_ns` evenly across the new word list. The FCPXML/`.uspxml` parser also runs a one-shot migration on load: if `seg.words[].text` joined by spaces no longer matches the normalized `seg.text`, it resyncs through the same helper, so existing projects with previously-edited subtitles get migrated word entries the next time they're opened (and persist on the next save).
- **Subtitle audio extraction no longer pulls samples from before the clip's source-in time**: `extract_audio_for_stt` did `pipeline.set_state(Paused)` (which prerolls a buffer at PTS 0) and then issued a `FLUSH | ACCURATE` seek to `source_in_ns`. If the preroll buffer survived the flush — or if the audio decoder streamed any samples ahead of the seek target before stabilizing — Whisper's "0s = first word" mapped to a file position earlier than the clip's actual source-in. The resulting subtitle timestamps then played about a sentence ahead of the audio when overlaid back at clip-local 0. The pull loop now inspects every buffer's PTS, drops or trims any samples whose PTS is below `source_in_ns`, and logs the first buffer's PTS vs the requested target so future mismatches surface immediately in the logs.
- **Window close button no longer needs a second click**: the `connect_close_request` handler called `win.close()` synchronously from inside `on_continue` (which itself runs synchronously when the project is not dirty). GTK4's `gtk_window_close()` emits `close-request` synchronously, so the handler re-entered itself: the inner invocation saw `close_approved=true`, ran proxy cache cleanup, and returned `Proceed`, but the outer invocation then returned `Stop` — and the outer Stop overrode the inner Proceed for the original close emission, leaving the window open. The user had to click the close button a second time before the deferred close actually fired. The handler now defers `win.close()` to the next idle via `glib::idle_add_local_once`, so the original `close_request` returns Stop cleanly and a fresh `close_request` from the idle callback proceeds normally. The duplicate `cleanup_for_unload` call in the early-return branch of the handler is also gone — cleanup now runs exactly once per close.
- **Tracker-bound (and title/adjustment) clips no longer drift horizontally between preview and export**: the export's keyframed-clip filter chains hard-coded the overlay position formula to the *normal* `(W-w)*(1+pos_x)/2` math, regardless of whether the clip used direct canvas translation. The preview, on the other hand, calls `direct_canvas_origin()` for clips with `tracking_binding`/`Title`/`Adjustment` kinds, which uses `(W-w)/2 + pos_x*W/2` (`= (W*(1+pos_x)-w)/2`). The two formulas give different positions for any non-zero `pos_x` (~10 % of canvas at `pos_x = 0.25`, scale 0.8), so a tracker-bound PNG ended up at one horizontal position in the program monitor and a different one in the rendered MP4. Both keyframed branches in `media::export` (primary base-track chain and secondary overlay chain) now check `clip_uses_direct_canvas_translation(clip)` and emit the matching direct-mode overlay expression. The non-keyframed branch already routed through `build_scale_position_filter` which handled this correctly. New unit test pins the predicate.
- **Static PNGs no longer flicker invisible when scrubbing the playhead**: `ProgramPlayer::seek_slots_in_place` (the paused-scrub fast path) was flushing the compositor and then per-decoder reseeking every slot — including stills.  Still-image source decoders are parked at EOS into `imagefreeze`, and the per-decoder reseek there is racy: sometimes `imagefreeze` re-emits its cached buffer cleanly after the compositor flush, sometimes it doesn't, leaving the still's compositor pad empty for the rest of that scrub.  Stills (and animated SVGs that have not been pre-rendered) are now skipped in the per-slot reseek loop, matching the existing fix in `play()` and `flush_compositor_for_still_refresh`.  The compositor flush alone is enough to make `imagefreeze` re-push its parked buffer when the compositor pulls the next aggregate.
- **Export crop now matches preview when the export resolution differs from the project resolution**: clip crop values are stored in *project pixels*. The preview's alpha-crop probe correctly scales them by `(buffer_dim / project_dim)` so the same fraction of the canvas is cropped at any preview-quality divisor. The export filter chain, however, was passing crop values straight into ffmpeg's `crop=…` filter as if they were already in *export-target* pixels — so a `crop_left = 153` on a 1920×1080 project (8 % of canvas) became `153 / 3840 = 4 %` of canvas when exporting at 4K, cutting only half as much off each edge as the preview showed.  `build_crop_filter` (and the keyframed crop expression generator) now also accept `project_w/h` and scale every crop value by `(out_dim / project_dim)`, so a 4K export of a 1080p project applies `153 × 2 = 306 px` from the left, matching the preview's proportional crop.  Two new tests cover the scaling and the resolution-match passthrough.
- **Chroma key on a still PNG no longer makes the whole image disappear**: the chroma-key chain inserted a `videoconvert` *before* the `alpha` element but nothing *after*.  The `alpha` element with `method=custom` outputs whatever format negotiation hands it, and the next downstream cap constraint is `capsfilter_zoom` (RGBA at project resolution).  For continuously-decoding video the negotiation eventually settles, but for a still PNG — where `imagefreeze` only ever pushes its single cached buffer — a failed negotiation leaves the slot's compositor pad empty for the entire playback and the still vanishes.  The chain is now bracketed by `conv_ck` (input convert) **and** `conv_post_ck` (output convert) so format negotiation through the alpha element stays stable for both stills and video.  The chroma-key live-update path also now uses the lighter `flush_compositor_for_still_refresh` instead of `reseek_slot_for_current` when the selected clip is a static image, so dragging the tolerance/softness sliders on a PNG no longer flushes the compositor with a racy per-decoder reseek.
- **Static PNGs no longer disappear from the Program Monitor when playback starts**: `ProgramPlayer::play()` was issuing a `FLUSH | ACCURATE` seek on every existing slot's decoder when transitioning from paused to playing — including stills.  A still-image source decoder is parked at EOS into `imagefreeze`, and the per-slot reseek is racy: it can clear `imagefreeze`'s cached buffer without managing to pull a fresh one from the parked decoder, leaving the still's compositor pad empty for the entire playback while the surrounding video tracks render normally.  Stills (and animated SVGs that have not been pre-rendered to a video file) are now skipped in `play()`'s reseek loop so `imagefreeze` keeps re-emitting its cached buffer once the pipeline is in PLAYING state.
- **Export crops past 500 px no longer get silently truncated**: `build_crop_filter` and the other `build_keyframed_property_expression` callers in `media::export` were passing literal `0..500` (crop), `±1.0` (position), `0.1..4.0` (scale), and `±180` (rotate) bounds as `min_value`/`max_value` arguments — `build_keyframed_property_expression` then clamped each per-frame keyframe to those bounds while building the ffmpeg filter expression, silently truncating values past the new ranges in the encoded output even though preview rendered them correctly.  All sites now pass the constants from `model::transform_bounds`, so the export ffmpeg graph honours the same crop/position/scale/rotate ranges as the inspector slider, the model-level keyframe clamp, the runtime evaluators, and the live transform-drag path.
- **Crop values past 500 px no longer snap back the moment playback starts**: the runtime crop evaluators in `ProgramClip::crop_*_at_timeline_ns`, the export keyframe evaluators in `media::export`, and the timeline-side `evaluate_clip_transform_at` helper still capped each crop edge at 500 pixels — the old slider ceiling, before this release bumped it to 4000. The user-facing slider, the model-level keyframe clamp, and the live transform-drag path were all updated to 4000 already, but the per-frame runtime evaluators were not — so dragging a crop to e.g. 800 px stored 800 in the model and rendered 800 while paused (the drag path bypasses the evaluators), then `apply_keyframed_video_slot_properties` ran on the first playback tick and re-fed the slot a clamped 500 via `crop_*_at_timeline_ns`, silently overwriting the user's value. Same story on the transform overlay's green crop box (read through `evaluate_clip_transform_at`) and on export. All twelve sites now clamp to 4000, matching the rest of the stack.
- **Underlying video tracks no longer blank out while transforming a still on top of them**: the still-image refresh path now also re-seeks every *non-still* decoder in the compositor after flushing it, so video tracks underneath a PNG keep showing their current-playhead frame while the user drags the PNG's transform/crop. Previously the compositor flush stripped every sink pad's queued buffer, leaving the underlying video pads empty until the next playhead move because their decoders are paused and don't push buffers spontaneously. The still's own decoder is intentionally left alone — it's parked at EOS into `imagefreeze` and the cached buffer survives the flush, so the PNG comes back via the same aggregate cycle as the freshly re-seeked video. The reseeks are fire-and-forget (no blocking `wait_for_compositor_arrivals`), so the GTK drag handler still returns immediately.
- **Tracker-followed PNG overlays no longer disappear during transform edits**: same root cause as the underlying-video blanking — the still-image refresh path now re-seeks the surrounding decoders, which keeps the entire composite (still + underlying tracks) on screen while the user is editing the PNG.
- **Attaching a tracker no longer snaps off-canvas position values back to ±1**: `resolve_translation_keyframes` in `src/media/tracking.rs` was clamping the resolved per-frame position keyframes to ±1.0, silently truncating any base position outside that range. The clamp now matches the renderer / inspector / model bound of ±3.0, so a clip with position values past the canvas edges keeps that placement when a tracker is attached.
- **Static PNG transform/crop drags no longer blank the Program Monitor**: still-image transform edits now skip the per-decoder reseek (which racily flushed the whole compositor and blocked up to 3 s waiting for a fresh aggregate from a decoder that had already EOSed into `imagefreeze`) and instead nudge the compositor with the same fire-and-forget flush used by title text/style updates. Selecting a still while paused now also pushes the still's own preview-inset to the transform overlay and forces a paused-frame refresh, so the Program Monitor immediately shows the selected PNG with correctly aligned handles instead of leaving the previous clip's frame on screen. As a defensive belt-and-braces, `clip_to_program_clips` now also extension-sniffs `is_image` for video-kind clips so a still that lost its `ClipKind::Image` on a hand-edited project still goes down the still path (with a warning logged on mismatch).
- **Static PNG/still-image clips no longer disappear during Program Monitor playback or transform refreshes**: normal still-image playback now stays on the existing safe image-preview path unless the clip is actually following a tracker, live preview reseeks pin static images to their source-in frame instead of advancing through nonexistent media time, and still-image transform drags now force a paused-frame refresh, so PNG overlays remain visible while playing or editing transforms.

### Performance
- **Fix Smooth-mode playback producing ~1 fps instead of 30 fps**: The `should_drop_late_for_playback_mode` function enabled leaky-queue + QoS drop-late policy for single-slot Smooth playback. This removed all backpressure from the compositor, causing it to spin at ~11,000 fps; the QoS sink (40ms max-lateness) then dropped 99.99% of frames, displaying only 1-2 per second. Fix: only enable drop-late during active transition overlaps or 3+ tracks, preserving backpressure-based 30 fps throttling for normal playback. Compositor now outputs 30.2 fps in Smooth mode (matching Balanced/Accurate).
- **Program preview boundary stutter reduction**: Three targeted optimizations that reduce clip-boundary stutter:
  - Enable `realtime_preview` by default — the incremental boundary path (hide departing, add entering) is now the default instead of requiring manual opt-in, eliminating the full teardown/rebuild cycle for most boundary crossings.
  - Reduce `teardown_single_slot` state-query timeout from 100ms to 10ms per element — after FlushStart, Null transitions complete near-instantly; the old 100ms×7 timeout blocked up to 700ms on the main thread.
  - Reduce `adaptive_arrival_wait_ms` floor from 200ms to 100ms — single-track sequential playback with pre-warmed sources needs shorter link waits.
  - Net result: 10s playback advance improved from 9549ms (4.5% loss) to 10017ms (~0%), average seek time reduced 41%, worst-case seek reduced 42%.

### Performance
- **Adjustment layer LUT optimization**: Adjustment layer preview probe pre-computes per-channel 256-entry lookup tables when overlay parameters change, reducing per-pixel work from 15 floating-point ops to 3 table lookups (fast path: ~1.2ms/frame at 1080p). Saturation is handled as an optional second pass only when != 1.0. LUTs are rebuilt once on slider change, not per frame.
- **Dev build opt-level 1**: Added `[profile.dev] opt-level = 1` to Cargo.toml for real-time preview performance in debug builds. Reduces adjustment probe per-frame cost from ~43ms (opt-level 0) to ~1.2ms (opt-level 1) at 1080p.
- **Non-blocking title editing**: Title text, position, and style changes no longer block the GTK main thread. GStreamer textoverlay properties are set instantly; the compositor flush is fire-and-forget (no spin-wait), and rapid changes are debounced via a 32ms glib timer so only one compositor flush fires per burst of edits. Previously each keystroke blocked for 15–500ms in a `wait_for_compositor_arrivals` spin-wait loop.
- **Lightweight MCP title updates**: The `set_clip_title_style` MCP tool now uses the same lightweight property-set + compositor-flush path as the UI instead of triggering a full pipeline rebuild. Reduces per-call latency from ~121ms (pipeline rebuild) to ~6ms (property set + async flush).
- **Responsive playhead scrubbing**: Interactive timeline scrubbing now caps compositor and decoder wait timeouts based on scrub velocity. Rapid seeks (< 300ms apart) use a 30ms arrival wait budget and 30ms playing-pulse timeout instead of the previous uncapped 500ms+ waits. The `should_prioritize_ui_responsiveness()` guard was incorrectly gated on `!background_prerender`, which is typically `true` — now applies to all paused seeks. Reduces average per-seek latency from ~237ms to ~90ms (2.6× faster), with no seeks exceeding 163ms (was 593ms).
- **Blend modes in background prerender**: Blend mode is now hashed in the prerender signature and applied in the ffmpeg prerender filter chain (using `blend=all_mode=` for non-Normal modes, `overlay` for Normal). Previously, prerendered segments always used `overlay` compositing regardless of the clip's blend mode setting.
- **Background prerendering includes title clips and effects**: Title clips now participate in the background prerender system. A timeline with 2 video clips and 1 title clip overlay (3 active clips) triggers prerendering. Title clips use `lavfi color` source (solid-color background) with `drawtext` filter (text, outline, shadow, bg box) in the ffmpeg prerender pipeline, matching the export-pipeline behavior. Previously title clips had empty source paths which would cause ffmpeg errors. Frei0r effects are now also included in prerendered segments (previously missing). The prerender signature hash now covers all title properties (text, font, color, position, outline, shadow, bg box, secondary text) and frei0r effects (plugin name, params, enabled state), ensuring stale prerendered frames are correctly invalidated when titles or effects change.

### Fixed
- **Adjustment layer sliders not updating preview**: Color/opacity slider changes on adjustment layers updated the internal overlay data but did not flush the compositor, so the Program Monitor stayed frozen until the playhead moved. The early-return path in `update_current_effects()` and `update_current_opacity()` now calls `reseek_slot_for_current()` when paused to force a compositor flush and re-process through the adjustment probe.
- **Adjustment layer causing pipeline rebuild failure**: `build_slot_for_clip()` returns `None` for adjustment layers (they have no GStreamer slot). The boundary-rebuild and `rebuild_pipeline_at()` loops treated this as a build failure, triggering full teardown/retry cycles on every frame tick. Now all slot-build call sites skip adjustment clips before attempting slot creation, and adjustment clips are excluded from the boundary clip-set comparison to avoid spurious rebuilds at adjustment layer start/end boundaries.
- **Adjustment layer frei0r effects not applied on export**: Two bugs: (1) adjustment clips were included in `primary_clips` / `secondary_clips_flat`, causing transparent placeholders to be concatenated into the video sequence (diluting the timeline with blank frames); fixed by filtering adjustment clips out at collection time and using a separate `all_adjustment_clips` list gathered before filtering. (2) The split→blend pattern for applying adjustment effects (`blend=all_mode=normal:all_opacity=1`) produced near-identity output because FFmpeg `blend` in normal mode doesn't output 100% of the top input; replaced with direct filter application for full-opacity adjustments (`[prev]{effects}[next]`), and overlay+colorchannelmixer alpha for partial-opacity adjustments.
- **Export filter_complex error with adjustment layers**: The adjustment effect chain had a spurious leading comma (e.g. `[vadj0work],frei0r=...`) because the per-filter builder functions return strings prefixed with `,`. FFmpeg parsed this as an empty filter name. Fixed by stripping leading/trailing commas from the concatenated effects chain.
- **Frei0r effects crash on parameter change (cairogradient, etc.)**: Changing any numeric slider on a frei0r effect that also had string parameters (e.g. cairogradient's `blend-mode`/`pattern`) caused a SIGSEGV. Root cause (from core dump analysis): `update_frei0r_effects` re-set ALL parameters on every slider change, including unchanged string params. The cairogradient frei0r C plugin crashes in `f0r_set_param_value` → `strlen` when a string property is re-set on a live GStreamer element (null/invalid pointer dereference inside the plugin). Fixed with three layers: (1) **skip string params in live updates** — the live slider path now only sets numeric params; string params are baked into the element during pipeline construction; (2) **detect string param changes** — if a string param actually changes (e.g. dropdown selection), compare against previous values before syncing and force a full pipeline rebuild (fresh element with new values) instead of a crash-prone live `set_property`; (3) **deferred reseek** — the blocking flush-seek + 3s wait was moved out of `update_frei0r_effects` into a debounced 32ms `glib::timeout_add_local_once` callback to avoid blocking the GTK main loop during slider callbacks. Additionally, all `set_property` calls are wrapped in `catch_unwind` and numeric values are clamped to the GStreamer `ParamSpec` range as defense-in-depth.
- **Title text size, position, and shadow mismatch between preview and export**: Three fixes: (1) Font scaling used `proc_h` instead of `project_h`, halving text size at Half quality. Fixed by using `project_height`. (2) Position used `halignment=position` + `xpos`/`ypos` which caused text-width-dependent clipping differences at lower resolutions. Replaced with `halignment=center` + `valignment=center` + `deltax`/`deltay` pixel offsets computed from `(rel - 0.5) × proc_dim`, plus a `text_h × 0.35` vertical compensation for textoverlay's top-edge center alignment. (3) Drop shadow now uses a dedicated shadow textoverlay element rendered behind the main text with matching font, shadow color (ARGB), and pixel-scaled offset (`shadow_offset × scale_factor`), replacing GStreamer's fixed ~1px built-in shadow. MCP-driven calibration shows size ratio avg=1.01, X error avg=0.2%, Y error avg=1%.
- **Title overlay edits not updating preview on non-current clips**: Editing title text, position, font, color, outline, shadow, or bg box on a clip that wasn't the highest-zorder "current" clip had no effect on the preview. The `on_title_changed` and `on_title_style_changed` callbacks used `current_clip_idx()` (highest-priority active clip) instead of the timeline-selected clip ID. Now both callbacks read `selected_clip_id` from timeline state and use `update_title_for_clip()` / `update_title_style_for_clip()` to target the correct slot by clip ID, matching how `update_transform_for_clip()` already works.
- **Frei0r effects crash on apply (medians, analog, cairogradient, etc.)**: Several frei0r plugins report NaN, Inf, or extreme bounds (f64::MIN/MAX) for their GStreamer parameter specs. These non-finite values were passed to GTK's `Scale::with_range()` which panics on NaN step increments. Fixed in two layers: (1) the frei0r registry now sanitizes min/max bounds during plugin discovery via `sanitize_param_bounds()`, clamping non-finite or extreme values to a sane 0.0-1.0 range, and (2) the inspector slider creation has a defensive fallback that validates bounds before constructing the Scale widget.
- **Patch-save path silently lost ~30 clip fields**: The incremental patch-save path (used when re-saving imported FCPXML files) only updated a subset of clip attributes (transform, color, title, keyframes). Blend mode, speed, reverse, freeze-frame, all grading extensions (exposure, black point, per-tone warmth/tint), chroma key, LUT path, frei0r effects, transitions, denoise, sharpness, flip, crop, volume, pan, audio settings, and color labels were never patched — changes to any of these fields were silently lost on save. The patch path is now disabled for dirty projects; all saves use the full rewrite which correctly persists every field.
- **Title clips generated broken asset references**: The FCPXML resource writer iterated all clips including title clips (which have empty source paths), producing `<asset>` elements with invalid `src="file://"` URIs. Title clips are now skipped in the asset-writing loop.
- **Incomplete known-attribute lists in writer and parser**: Added ~20 missing field names (`us:blend-mode`, `us:exposure`, `us:black-point`, all title styling fields, `us:bg-removal-*`, `us:clip-kind`) to both `is_writer_managed_asset_clip_attr` and `is_known_asset_clip_attr` so they're properly categorized instead of being treated as unknown passthrough attributes.
- **Deleted clips reappear after save/reopen**: The patch-save path (`patch_imported_fcpxml_transform`) iterated over current project clips and patched their attributes in the original XML, but never removed `<asset-clip>` blocks for clips that had been deleted. Deleted clips survived in the original XML verbatim. Now the patch path detects clip count mismatches (additions or deletions) and falls through to the full rewrite path, which only writes the current project state.
- **Frei0r effects lost on project reopen**: The FCPXML writer manually escaped `"` → `&quot;` in the frei0r effects JSON, but quick_xml then also escaped `&` → `&amp;`, producing double-escaped `&amp;quot;` in the file. On parse, `unescape()` only decoded one level, leaving `&quot;` in the JSON string, causing `serde_json::from_str()` to fail silently (empty vec via `unwrap_or_default()`). Fixed by removing the manual quote escaping from the full-write path (quick_xml handles it), and adding backward-compatible `&quot;` → `"` decode in the parser for existing files.
- **Title styling controls now live-update preview**: Outline width/color, drop shadow toggle/color, background box toggle/color/padding, font button, and text color button all update the GStreamer textoverlay element immediately. Previously these 11 widgets had no signal handlers — changes only took effect after a full project reload.
- **Title style applied on slot reuse**: The fast-path slot reuse code (`update_slot_effects`) now also calls `apply_title_style_to_slot`, ensuring outline/shadow/bg_box properties are applied when the compositor reuses a decoder slot at boundary crossings.
- **Export audio DTS error with title clips**: Timelines containing title clips (or any video-only sources) could cause `amix` + `adelay` to produce trailing audio packets with `PTS=NOPTS_VALUE` (INT64_MAX), triggering "non monotonically increasing dts" muxer errors. Fixed by adding `atrim=duration=<project_dur>` after `amix` to ensure the audio output is cleanly bounded to the project duration. The lavfi color source for title clips also now uses explicit `trim` + `setpts=PTS-STARTPTS` for clean monotonic timestamps.
- **Title overlay preview/export size parity**: Title text now renders at the same visual size in both GStreamer preview and FFmpeg export. Font sizing uses a resolution-independent reference height (1080p) — Pango points are converted to pixels (×4/3) and scaled proportionally to the actual rendering height. Position formulas now use center-at-position semantics in both pipelines (`x = rel_x*w - text_w/2`) to match GStreamer textoverlay behavior. Minimum font size floor of 4pt prevents illegible text at very small preview sizes.
- **Overlay clip frame sync on export**: Secondary/overlay track clips were missing the `fps` filter that primary clips have, and carried residual PTS offsets from keyframe seeking. Both paths now normalize to the project frame rate before compositing, and overlay PTS is reset via `PTS-STARTPTS` before applying the timeline offset to eliminate seek-induced drift.
- **Clip trim lost on USPXML reopen**: The FCPXML patch-save path updated `offset` and `start` attributes but not `duration`, so trimmed clip lengths reverted to their original value when reopening a saved project.
- **Frei0r effects JSON breaks FCPXML parsing**: Saving a project with frei0r effects produced invalid XML — the `us:frei0r-effects` attribute contained raw JSON with unescaped `"` characters that broke XML attribute boundaries. Now escapes quotes to `&quot;` on write. Existing files with unescaped JSON are handled via backward-compatible sanitization on load.
- **Frei0r export boolean parameters**: FFmpeg frei0r wrapper requires boolean params as `y`/`n`, not `1.000000`/`0.000000`. Filters with boolean parameters (e.g. `3-point-color-balance` split-preview, `alpha0ps` display-input-alpha) now export correctly.
- **Frei0r registry caching**: Plugin discovery (dlopen of ~95 .so files) now runs once via `OnceLock` cache instead of on every access. Prevents crashes when repeated dlopen/f0r_init calls conflict with active GStreamer pipelines.
- **Match Clip Colors**: Automatically grade one clip to match the color appearance of a reference clip. Uses Reinhard-style statistical color transfer in CIE L\*a\*b\* space — samples representative frames from both clips, computes per-channel mean and standard deviation, and maps deltas onto existing clip color parameters (brightness, contrast, saturation, temperature, tint). Per-zone histogram analysis classifies pixels by luminance into shadow/midtone/highlight zones and estimates residual grading adjustments (shadows, midtones, highlights brightness, per-zone warmth/tint, and black point) beyond what the global sliders express. When the reference clip has existing color grading (sliders or LUT), the match targets the graded appearance rather than the raw source. Optionally generates a 17³ 3D `.cube` LUT for fine-grained non-linear residual matching beyond what sliders can express — the LUT captures only the difference between slider-adjusted source and graded reference, avoiding double-application. Available via: **Inspector** "Match Color…" button (dropdown to pick reference clip + optional LUT checkbox), **keyboard shortcut** `Ctrl+Alt+M`, and **MCP tool** `match_clip_colors`. Fully undoable — restores all 17 color parameters and LUT path in one step.
- **Copy & Paste Color Grade**: New dedicated color-grading copy/paste via `Ctrl+Alt+C` / `Ctrl+Alt+V`. Copies only color-related properties (brightness, contrast, saturation, temperature, tint, exposure, black point, shadows/midtones/highlights, per-tone warmth/tint, denoise, sharpness, LUT path) as static values between clips. MCP tools `copy_clip_color_grade` and `paste_clip_color_grade` added for automation.

- **Idle background prerender scheduling + status bar integration**: Background prerender now also queues nearby complex sections while paused/stopped (throttled around the playhead), and the existing bottom proxy progress bar now reports prerender activity (or combined proxy+prerender activity).
- **Background disk prerender for complex overlap sections**: Added an optional background prerender mode that renders upcoming 3+ track overlap windows to temporary disk clips and uses them when available at rebuild boundaries, with automatic fallback to live rebuilds when cache misses occur.
- **Boundary rebuild debounce**: Playback boundary handling now debounces duplicate full-rebuild attempts for the same desired clip set within a short window (~120ms), reducing rebuild thrash during transient overlap churn.
- **Realtime prewarm gating**: Pre-preroll sidecar prewarming is now constrained to active playback near imminent boundaries and torn down outside the boundary window, reducing steady-state realtime-preview overhead.
- **Reuse-path matching expansion**: Same-count decoder reuse now matches desired clips to reusable slots by compatible source/effects topology (not only positional index) and re-applies compositor z-order from desired track order.
- **Occlusion hot-path probe reduction**: Audio-only occlusion/reuse hot paths now use fast cached audio-presence lookups (defaulting non-blocking when unknown) to avoid blocking Discoverer probes in tight rebuild loops.

The VA-API decoder outputs GPU memory (`video/x-raw(memory:VAMemory)` / DMABuf) which cannot be CPU-mapped for the `videoconvertscale`/`videobalance` effects chain. Five fixes: (1) lowered VA-API decoder ranks (`vah264dec`, `vah265dec`, etc.) to `MARGINAL` so GStreamer auto-plugging selects software decoders (`avdec_h264`, `avdec_h265`) that output plain `video/x-raw`; (2) added a leaky queue (`max-size-buffers=2, leaky=downstream`) after the prescale capsfilter to decouple the decode/prescale thread from the effects/display chain; (3) source preview now checks for and uses proxy files when available, and auto-requests proxy transcodes for video media loaded into the source monitor; (4) added GStreamer bus error monitoring on the source preview pipeline so decoder/negotiation errors are logged instead of silently freezing; (5) fixed STATE_LOCK deadlock in `Player::play()`/`pause()`/`stop()` where `set_state()` could block indefinitely if the pipeline had a pending async state change — state transitions now defer via a repeating timeout when async work is in progress.
- **Source preview low FPS on high-resolution media**: The source preview effects chain processed video at 1920×1080 even though the widget is typically ~320×200 pixels (~30× more pixels than needed). Four optimizations: (1) adaptive prescale — default resolution lowered to 640×360 with runtime updates to 2× widget size via a 100ms polling timer; (2) removed `gaussianblur` from the filter chain — it only accepts AYUV format, forcing two expensive I420↔AYUV format conversions per frame even when sigma=0 (neutral); denoise/sharpness is preserved during export; (3) eliminated all 4 `videoconvert` elements from the chain — `videocrop`, `videobalance`, `videoflip` all accept I420 natively; (4) added `format=I420` hint to the prescale capsfilter to prevent random RGBA format negotiation. Combined improvement: ~50× reduction in per-frame CPU work at typical widget sizes with neutral effects.
- **Source preview smoothness under load**: Added a playback-only drop-late policy in the source player (`queue leaky=downstream` + tighter buffering + sink QoS/max-lateness while Playing, restored to non-leaky conservative buffering when Paused/Stopped). Source proxy selection is now adaptive when proxy mode is Off (Quarter for small source-monitor sizes, Half for larger), and proxy encoding is tuned for faster decode (`-tune fastdecode`, no B-frames, 1 reference frame, `yuv420p`) to improve preview FPS on CPU-bound media.
- **Adaptive VA-API source decode with safe fallback**: Source preview now detects VA decoder availability and, when hardware acceleration is enabled, prefers a hardware-fast decode mode for smoother playback in `cargo run`/Flatpak. If the hardware path fails (e.g. negotiation/DMABuf/VA errors), the player automatically falls back to the software-filtered decode mode and reloads the current source URI, preventing persistent black/frozen playback.
- **Source monitor seek responsiveness parity improvements**: Added frame-boundary deduplication for paused Source Monitor seeks (skips redundant same-frame re-seeks during scrubbing), plus a dedicated Source Monitor playback-priority setting (`smooth`/`balanced`/`accurate`) exposed in Preferences and MCP (`set_source_playback_priority`). Smooth/Balanced use keyframe seeks for lighter scrub/playback behavior; Accurate preserves frame-precise seeks.
- **Static-like preview after Video 4 clip exits (3→2 transition)**: Incremental boundary fast paths could leave transition updates visually stalled while audio continued. Boundary transitions now use the proven full rebuild path until incremental add/remove correctness is fully hardened.
- **Transition preview frame regression after incremental add path**: Add-only incremental boundary handoff could miss clip-enter visual updates in some transitions and with half-sized proxy preview in the three-track sample. The playback boundary path now uses the proven full rebuild for clip-enter transitions.
- **Incremental boundary slot diffing correctness**: Re-enabled incremental add-only and remove-only boundary paths with 5 bug fixes: (1) reset `comp_arrival_seq` on retained slots before snapshotting so `wait_for_compositor_arrivals` requires genuinely fresh post-seek buffers; (2) use `FLUSH|ACCURATE` seeks during playback to prevent long-GOP keyframe snap-back on proxy media; (3) extracted shared `seek_all_slots`/`reset_slot_arrival_seqs` helpers; (4) wire `teardown_prepreroll_sidecars()` and `record_rebuild_duration_ms()` for proper cleanup and adaptive wait telemetry; (5) use adaptive arrival wait budgets matching the full rebuild path. Falls back to full rebuild automatically if incremental path cannot handle the transition.
- **Incremental boundary paths disabled (retained-decoder regression)**: Incremental add/remove boundary paths are disabled behind a compile-time constant while a retained-decoder frame-refresh regression is investigated. After an incremental update, retained decoders stop producing new frames during playback, leaving the compositor stuck on a single static frame while audio continues. The fixed methods are preserved for re-enabling once the root cause (likely compositor aggregator segment/running-time mismatch on retained pads after flush+seek) is resolved.
- **Incremental boundary paths — frame drop and screencast rendering fix**: Fixed two bugs in the "no-touch-retained" incremental strategy: (1) add-only path didn't seek retained decoders or reset start_time, causing new decoders' running-times to be misaligned (frames dropped as "late" by the sink — screencast video never rendered); (2) remove-only path removed elements while the pipeline was still Playing, risking streaming thread races. Both paths now pause the pipeline, seek ALL decoders individually (per-decoder FLUSH only propagates through its own branch — no compositor-level flush), reset start_time, and wait for all preroll + arrivals before resuming.
- **Incremental boundary add-only disabled (aggregator state regression)**: Disabled the add-only incremental path after confirming that individual per-decoder FLUSH seeks don't trigger GstVideoAggregator's coordinated seek handler — the compositor's internal segment/position tracking stays stale, leaving retained decoders' video frozen. Root cause: only `compositor.seek_simple` (src-pad seek) properly resets aggregator state, but it propagates upstream to ALL decoders causing double-flush corruption. Remove-only incremental path remains enabled (safe — no seeking/flushing of retained decoders). Add-only transitions use the proven full rebuild.
- **Choppy playback after remove-only incremental transition**: Remove-only incremental paths recorded their fast rebuild times (~100ms) into the adaptive wait ring buffer, contaminating the p75 calculation. Subsequent full rebuilds (e.g. 2→3 track transitions) used dangerously tight preroll/arrival budgets (scale=0.6 instead of ~1.0), causing persistent frame drops that never recovered. Fix: remove-only incremental paths no longer record their duration in the telemetry ring buffer.
- **Remove-only incremental disabled (aggregator timing stale after pad removal)**: Disabled the remove-only incremental path after confirming retained decoders produce ≤1 frame/sec after compositor sink pads are removed. The aggregator's internal timing/segment state requires `compositor.seek_simple` to reset, which is the same fundamental GstVideoAggregator limitation that blocked the add-only path. All boundary transitions now use the proven full rebuild.
- **Occlusion-based video decode skip**: Clips fully occluded by an opaque, full-frame clip above them now build audio-only slots (uridecodebin with audio-only caps), completely skipping video decode, effects processing, and compositor pad allocation. Reduces CPU/GPU load and boundary rebuild time in multi-track projects where lower tracks are hidden behind a full-frame overlay.
- **Audio ending early vs video during transition-heavy preview playback**: While active video-slot rebuilds were running at clip boundaries, the audio-only playbin could continue advancing independently, causing cumulative A/V drift (audio finishing early) in multi-track projects. Playback-boundary rebuilds now pause and re-sync the audio-only pipeline to the current timeline position before resuming playback, keeping audio and video end timing aligned.
- **Intermittent project-open seek freeze (`futex_wait`)**: Hardened reload/seek state transitions to avoid pipeline-wide `Ready` resets in the hot path: `ProgramPlayer::load_clips()` now keeps the main compositor pipeline in `Paused` after slot teardown, and `rebuild_pipeline_at()` now resets `start_time` instead of forcing `pipeline.set_state(Ready)`. This avoids pad-deactivation lock contention seen when opening a project and seeking immediately.
- **UI lock-up risk during paused 3+ track seeks**: Paused preroll/arrival waits are now budget-capped in responsiveness mode, and rebuild fallback second-pass settle is skipped when it would exceed the UI budget. Added seek/rebuild phase timing logs to make remaining stalls diagnosable.
- **Autoplay after project load/new**: Program Monitor reload now suppresses playback resume for full project replacement actions (Toolbar New/Open/Recent and MCP `open_fcpxml`/`create_project`), so loading a project no longer starts playback automatically.
- **Preview frames advancing on project open**: Opening a project caused visible frame advancement in the Program Monitor (without audio) because the compositor playing pulse briefly set the pipeline to Playing to flush preroll. The initial seek after project load now skips the playing pulse entirely and transitions the player state from Stopped to Paused, relying on Paused-state preroll to display the first frame.
- **Transform overlay background playback while dragging**: Starting a transform/crop drag in the Program Monitor now pauses timeline playback immediately, and transform live mode keeps the pipeline paused so the frame does not keep advancing in the background during edits. Releasing the drag still performs the final reseek refresh.
- **Audio-less clips stalling audiomixer**: Clips without an audio stream (e.g. screencasts, image sequences) still had an audiomixer sink pad requested, leaving an unfed pad that could stall the aggregator. Fixed by setting `is-live=true` on the background `audiotestsrc` so the audiomixer operates in live mode (clock-paced, won't wait indefinitely for unlinked pads). Added `probe_has_audio_stream()` using GStreamer Discoverer to detect audio-less clips, and the slot builder now skips the `audioconvert → audiomixer` path entirely for such clips.
- **Third video track not rendered with 3+ active tracks**: When three or more video tracks were active at the playhead, the top-priority track's video was missing from the preview. After the two-pass compositor settle confirmed all decoder buffers had arrived, a redundant `reseek_slot_for_current()` flushed only the top slot's compositor pad, invalidating the compositor's preroll state. The subsequent async playing pulse could not reliably re-aggregate all pads within the 250 ms window. Removed the redundant reseek. Additionally, single-slot reseeks (used by transform drag-end refresh) now flush the compositor and reseek ALL decoder slots so the compositor can produce a complete composited frame from every video track.
- **Transform edits not applied to non-top-track clips**: `update_transform_for_clip` only pushed crop/rotate/flip/scale/position to GStreamer when the edited clip was the highest-priority (top-track) clip. Clips on lower tracks stored model changes but never applied them to the pipeline until a rebuild. The fix removes the `current_idx` guard so transforms are applied to any active slot, and also wires scale/position through `apply_zoom_to_slot` (previously ignored during live edits).
- **No preview refresh after transform overlay drag ends**: Added an `on_drag_end` callback that triggers a paused-frame reseek when the user releases the mouse after dragging crop, scale, or position handles. This ensures the final composited frame reflects the last transform state.
- **Crop not applied during export**: Added ffmpeg `crop` + `pad` filter to the export pipeline so per-clip crop values (left/right/top/bottom) are applied when exporting. Primary-track clips pad cropped regions with black; secondary-track (overlay) clips pad with transparent pixels so lower tracks show through.
- **Cropped clips showing black instead of lower tracks in preview**: Moved the GStreamer `videocrop` element from source resolution (before downscale) to project resolution (after RGBA conversion), and added a `videobox` re-pad step with `border-alpha=0` so cropped areas become transparent. The compositor now correctly reveals lower-track video through cropped regions of upper-track clips.
- **Video 2 (B-Roll) restarting when Video 4 enters/exits during preview playback**: Playback-boundary rebuilds used `KEY_UNIT` decoder seeks in smooth/balanced modes. On long-GOP proxies (e.g. keyframes at 0s and ~10.4s), seeks around 9–10s snapped back to the 0s keyframe, making B-roll appear to restart whenever a track-boundary rebuild occurred. The playing rebuild path now forces `FLUSH|ACCURATE` seeks so all active clips resume from the correct source position across 2→3 and 3→2 track transitions.
- **Clips restart from beginning at track boundaries during playback**: When a new clip entered or left the active set during playback (e.g., Video 4 starting), `rebuild_pipeline_at()` tore down all slots and rebuilt them, but the pipeline remained in Ready state when decoder seeks were issued. GStreamer decoders in Ready state silently reject seeks, so when the pipeline finally transitioned to Playing, decoders prerolled at position 0 — restarting all clips. The fix explicitly transitions the pipeline to Paused and waits for decoder preroll before seeking, ensuring seeks succeed.
- **Frozen compositor output across seeks (multi-track)**: The compositor produced identical (black) frames regardless of playhead position because stale preroll frames from the initial pipeline setup were never cleared. Per-decoder flush seeks reset individual compositor sink pads but left the compositor's aggregation state and downstream buffers untouched. The fix atomically flushes the compositor (`compositor.seek_simple(FLUSH, ZERO)`) before per-decoder seeks, clearing stale preroll so the compositor re-aggregates with fresh decoder buffers. Applied in `seek_slots_in_place`, `reseek_all_slots_for_export`, and the paused settle path of `rebuild_pipeline_at`. Additionally, `wait_for_compositor_arrivals()` spin-waits until post-seek buffers from every decoder slot have actually reached the compositor, preventing races where the playing pulse fires before decoder output propagates through the effects/queue chain.
- **Proxy cache infinite retry loop**: `ProxyCache::request()` re-enqueued failed proxy transcodes on every 250 ms poll cycle, causing 18 proxy requests for 3 clips. A `failed` set now permanently tracks failed transcodes and prevents re-queuing.
- **Proxies not used on initial project open**: Proxy path resolution now happens before `load_clips()` in the same deferred callback, so the first `rebuild_pipeline_at()` uses pre-rendered proxies immediately. Previously, proxy paths were populated in a separate deferred callback that ran after the pipeline was already built with original source files.
- **Program monitor deadlock on project change**: `teardown_slots()` now pre-flushes all compositor/audiomixer sink pads before removing decoder branches. Streaming threads could be blocked in downstream pushes (holding the pad STREAM_LOCK), preventing `set_state(Null)` from deactivating pads. The flush unblocks those threads first, eliminating the `futex_wait` deadlock that froze the UI when reloading clips (e.g. after moving a clip or editing properties during playback).
- **Black preview when scrubbing while paused**: Two fixes for the program monitor showing a black frame instead of the composited image when moving the playhead while paused:
  1. **Fast-path preroll wait**: `seek_slots_in_place` now waits for decoders to preroll *before* the Playing pulse. Flush seeks clear the compositor's input buffers; without this wait the Playing pulse fired before new decoded frames arrived, so the compositor had nothing to aggregate.
  2. **Paused rebuild after project changes**: `on_project_changed` now calls `seek()` when paused (not just when playing) so the pipeline is rebuilt with the correct composited frame. Previously, `load_clips()` tore down all decoder slots but skipped the rebuild when paused, leaving the monitor black until the user manually seeked.
- **Freeze with 3+ video tracks during paused seek**: `wait_for_paused_preroll()` now waits only on decoder elements instead of the full pipeline. `pipeline.state()` blocked the GTK main thread waiting for `gtk4paintablesink` to complete Paused preroll, but the sink needs the main loop for `gdk_paintable_invalidate_contents()` — deadlocking with 3+ tracks where decoding takes long enough that the sink hasn't prerolled before we block. Decoder-only waits avoid this while still ensuring frames are decoded at the compositor inputs; the display sink prerolls asynchronously when control returns to the main loop.
- **Playback stall with 3+ overlapping clips**: `rebuild_pipeline_at()` now waits for decoder re-preroll after flush seeks before entering Playing state. The flush seeks reset the internal multiqueue preroll, so the compositor entered Playing with empty input pads and stalled indefinitely (non-live GstAggregator). Additionally, unlinked decoder pads and failed seeks now send EOS to their compositor/audiomixer pads so the aggregator never waits for buffers that will never arrive.
- **Import-time `gstplaysink` abort mitigation**: Media import and external-drop paths no longer auto-select/auto-load the newly imported clip into the Source Monitor. Loading now happens on explicit user selection, avoiding import-time overlap between Discoverer probing and source `playbin` reconfiguration on problematic media.
- **Source import preview stability**: Hardened source-monitor `playbin` URI reloads by transitioning through `Null` before setting a new URI, and ignoring duplicate same-item selection callbacks. This prevents rapid reconfiguration races that can abort with `gstplaysink` assertions during import/selection.
- **Import abort in new projects (`gstplaysink` assertion)**: Import UI paths were invoking source selection twice (`select_child` callback + an explicit second callback), which could trigger overlapping `playbin` reconfiguration and crash with `gstplaysink.c:1475: try_element: assertion failed: (!element_bus)`. Import now selects the item once and relies on the existing selection callback to load preview media.
- **MCP `import_media` hang after project open**: Importing media through MCP no longer triggers `on_project_changed()`. Importing into the library is now library-only (matching UI behavior), which avoids an unnecessary program-player reload that could stall subsequent MCP commands.
- **Crash/hang during project updates**: `teardown_slots()` now detaches slot branch elements from the pipeline, transitions the removed elements to `Null`, and only then releases compositor/audiomixer request pads. This avoids both the prior race (in-flight audio query hitting a released pad) and the new hang observed when forcing `Null` while branches were still attached.
- **Playhead seek preview**: Seeking the timeline playhead while paused now correctly shows the frame at the target position instead of a black screen or the first frame of the clip. Three complementary fixes:
  1. **In-place seek fast path**: When the same clips are already loaded for the target position, decoders are seeked in-place (no pipeline teardown/rebuild). The previous always-rebuild approach went through GStreamer's `Ready` state (flashing black background) and allowed decoders to preroll at position 0 before the seek was applied (flashing first frame).
  2. **Rebuild path display pulse**: After a full pipeline rebuild (cold start, clip boundary crossing), a brief `Playing` pulse (150 ms) is applied before returning to `Paused`. Without this, the GStreamer compositor holds its composited output until the clock advances; the pulse releases that back-pressure so the frame actually reaches `gtk4paintablesink` and the GTK paintable is updated.
  3. **Fast-path display pulse**: The in-place seek fast path now also performs a `Playing → Paused` pulse after seeking decoders. Per-decoder FLUSH events stop at the compositor's sink pads and are not forwarded downstream, so the display sink stays prerolled with its old frame; the pulse starts the clock briefly to flush the new composited frame through to `gtk4paintablesink`.

This also prevents FLOW_NOT_LINKED errors from reaching the pipeline bus, which could corrupt the pipeline state and produce black video or audio static.
- **Black preview when scrubbing**: The faster teardown (above) eliminated implicit settling time that the old synchronous `set_state(Null)` calls provided. New decoders hadn't reached Paused state when seeks were issued, so seeks were silently ignored. Now `rebuild_pipeline_at` always waits for preroll when paused (scrubbing), only skipping the wait during playback boundary crossings to avoid stutter.
- **Timeline draw performance and loading freeze**: Optimized waveform rendering from O(n) individual `stroke()` calls per pixel to 3 batched strokes per clip (one per color band: green/yellow/red). Added a `loading` guard to the timeline that suppresses click and drag events while a project file is being parsed, preventing interaction before the timeline is ready. Capped concurrent thumbnail and waveform extraction threads at 4 each (previously unlimited — large projects could spawn 200+ simultaneous GStreamer pipelines). Also fixed off-screen clip culling to use actual widget width instead of a hardcoded 4000px limit.
- **Thumbnail/waveform cache deadlock**: Fixed a bug where failed background extraction threads (e.g., corrupted file, missing audio stream) never signalled completion, permanently consuming a concurrency slot. After 4 failures, both caches would stall entirely — no new thumbnails or waveforms could load for the rest of the session. Extraction threads now always signal completion regardless of success or failure.
- **Excessive per-frame waveform computation**: Waveform peak resampling (`get_peaks`) and Cairo path building were computed for the *full* clip width (potentially 60,000+ pixels for long clips) every frame, even when only ~1,000 pixels were visible. Now only the visible portion of each clip's waveform is computed and drawn, reducing per-frame work by 10–60× for scrolled or zoomed-out timelines.
- **Program monitor playhead freeze (compositor sync)**: Fixed compositor-based pipeline deadlocking during mid-playback clip boundary rebuilds. The always-on `videotestsrc` background accumulated running-time while newly-created decoders started at running-time 0 after flush-seek, causing the compositor to wait for decoders to catch up. Now the pipeline transitions through Ready state during rebuilds to reset the running-time base. Also fixed `audioconvert` element leak — per-slot audioconvert elements were not tracked or cleaned up during slot teardown, causing orphaned elements to accumulate across rebuilds.
- **Program monitor playhead freeze with 3+ video tracks** *(pre-compositor; superseded by compositor rewrite)*: Previously fixed in the old 3-playbin architecture by prerolling through PAUSED before seeking. Root cause was stale `query_position` values after pipeline hot-swap. Now structurally eliminated by the compositor rewrite which uses wall-clock position tracking instead of GStreamer position queries.
- **Export overlay transparency with 3+ video tracks**: Fixed secondary-track overlay clips with zoom-out (scale < 1.0) rendering opaque black borders instead of transparent padding during export. The `format=yuva420p` conversion was applied after the scale/position pad filter, so `black@0` had no alpha channel to work with. Reordered the filter chain so color/denoise/sharpen/LUT effects run in yuv420p first, then format conversion to yuva420p occurs before scale/position padding, making overlay borders truly transparent and revealing lower tracks beneath.
- **Program Monitor PiP live preview**: Made the top Program Monitor layer background transparent so per-pixel alpha from scale/position transforms can reveal the lower active video track during live preview.
- **Program Monitor regression fixes**: Restored +/-/Fit zoom behavior when baseline canvas size is not yet available by falling back to scroll-viewport/project dimensions, and improved PiP preview reveal by forcing transparent `videobox` borders in the zoom/position chain so lower-track underlay video can show through uncovered regions.
- **Program Monitor layered zoom alignment**: Both monitor layer pictures now force `halign/valign=Fill` and share the same preview CSS class, so B-roll/underlay content follows the same +/-/Fit zoom geometry as the primary layer while clips are moved.
- **Program Monitor underlay zoom-out floor**: Enabled `GtkPicture::set_can_shrink(true)` on both monitor layers so the underlay/B-roll layer can scale below 100% together with the main layer.
- **Program Monitor underlay zoom-out floor (follow-up)**: Set both monitor pictures to a minimal size request and excluded the top overlay picture from size measurement, preventing the underlay layer from clamping near 75% when zooming to 50%/25%.
- **Program monitor transform refresh**: Scale/Position edits now reliably refresh the

### Fixed
- **Media buttons (Append, Set In, Set Out) broken after first import**:
  - When a file is first imported, `duration_ns = 0` because the ffprobe runs in the background.
    `on_source_selected` set `source_marks.out_ns = 0`, which caused Append to create a
    zero-duration clip (silent no-op) and Set In to always clamp to zero regardless of
    scrubber position.
  - Clicking away and back on the item worked because by then the probe had completed and
    the correct duration was used.
  - Fix: the 100ms preview poll timer (which already queries `p.duration()` for the timecode
    label) now syncs `source_marks.duration_ns` and `source_marks.out_ns` from the player
    the first time a valid duration becomes available. The player pipeline prerolls in
    ~100–300ms, well before the user can click any buttons.
- **Proxy media not used on startup / LUT invisible in preview**:
  - `on_project_changed` called `cache.request()` (which synchronously adds disk-cached
    proxies to the in-memory map) but never pushed the result to the player.
    The player's `proxy_paths` map stayed empty so every clip fell back to its source
    file. Proxy paths are now forwarded to the player immediately after the request loop.
  - The 500 ms proxy poll timer only called `update_proxy_paths` when a background
    transcode completed (`resolved` non-empty). Synchronously-cached disk proxies were
    never reflected in `resolved`, so the player never learned about them. The timer now
    syncs proxy paths whenever the cache is non-empty and proxy mode is enabled.
  - Because the LUT is baked into the proxy file during ffmpeg transcode, the above fix
    also restores LUT visibility: once the correct LUT-baked proxy is used, the LUT is
    visible in the program monitor preview.
- **Playback stops on second (or later) cross-dissolve when clips have a 1-frame gap**:
  - `ns_to_fcpxml_time` uses integer frame-count division (floor), so clip positions
    can be off by 1 frame (≈41 ms at 24 fps) after an FCPXML save/load round-trip.
    `clip_at()` used exact `[start, end)` bounds, so a gap of even 1 ns between clip B
    and clip C caused the handoff to return `None` → the player treated it as
    end-of-timeline and stopped.
  - `activate_transition()` used `c.timeline_start_ns == clip_timeline_end_ns` (exact
    equality) to find the incoming clip — also broken by sub-frame gaps.
  - Fix: `clip_at()` now has a fallback that bridges gaps up to 100 ms ahead (≥2 frames
    at 24 fps) by finding the next-earliest clip starting in that window. This is safe
    for all existing call-sites (scrubbing, seeking, handoff detection).
  - `activate_transition()` incoming-clip search changed to a range check:
    `start_ns ∈ [clip_end_ns, clip_end_ns + 100 ms]`.
  - `transition_opacities()` now gates on `transition_active` so picture_b is never
    made partially visible when pipeline2 is not actually running.
- **Choppy playback around transitions**:
  - `activate_transition()` previously called `pipeline2.state(120ms)` — a blocking
    wait on the GTK UI thread. Because `poll()` runs from a 33ms GTK timeout, one
    120ms block dropped ~4 frames and caused a visible stutter at the start of every
    cross-dissolve.
  - Fix: removed the blocking wait. `activate_transition` now sets pipeline2 directly
    to `Playing` and records `pipeline2_pending_seek_ns`. The `seek_simple()` to
    `source_in_ns` is issued on the very next `poll()` tick (33ms later) by which
    point the pipeline is ready — zero UI thread blocking.
- **Cross-dissolve reverts to previous clip after transition**:
  - `load_clip_idx` applied `transition_alpha()` to the GStreamer `alpha` filter when
    the incoming clip loaded. At the clip boundary `timeline_pos = clip_B.timeline_start_ns`,
    `t = 0` → `alpha = 0.0`. GStreamer pipeline1 became fully transparent; GTK4 Picture
    retained clip A's last frame, appearing as if playback rewound.
  - Fix: `load_clip_idx` now always sets `alpha_filter.alpha = 1.0`. Cross-dissolve
    blending is handled entirely by `picture_a.set_opacity()` / `picture_b.set_opacity()`
    in the 33ms poll timer.

  - The seek flags used during scrubbing/paused seeks were `KEY_UNIT` (Smooth/Balanced
    priority), which snaps to the nearest keyframe before the playhead. For H.264 media
    with long GOP intervals this could display a frame seconds away from the actual
    playhead. Paused seeks now always use `ACCURATE` flags so the exact frame is decoded.
  - In **Smooth** playback priority mode, the preroll wait before seeking to a different
    source file was skipped. This caused the seek to be issued before the pipeline reached
    PAUSED, which GStreamer silently ignores, leaving the preview at frame 0 of the new
    clip. The preroll wait is now unconditional when not playing (150 ms cap).
  - The same `KEY_UNIT` bug also affected the Inspector's crop/rotation/flip controls
    and the title overlay position—adjusting these while paused would seek to the wrong
    frame. Fixed to use `ACCURATE` seeks in all non-playing seeks.
  - Inspector color/effects sliders (brightness, contrast, saturation, denoise,
    sharpness) did not visually update the preview when adjusted while paused. The
    GStreamer pipeline filter properties were updated correctly but the `KEY_UNIT` seek
    used to force a frame redraw could be a no-op when already at a keyframe boundary,
    leaving the preview stale. Fixed to use `ACCURATE` seeks so the current frame is
    always re-decoded with the new filter values applied.
